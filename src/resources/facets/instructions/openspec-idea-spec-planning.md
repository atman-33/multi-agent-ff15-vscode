# Idea-to-Spec Planning — Instructions

1. If the execution workspace is a git repository and an `origin` remote exists, run `git fetch origin --prune` first to retrieve the latest remote history.
2. When `origin/main` is available, inspect the latest code from there by moving to `origin/main` with `git switch --detach origin/main` before planning. When `origin/main` is not available, inspect the current local branch or `HEAD` instead and continue without blocking on the missing remote.
3. Read `{{ output("clarify-requirements", "latest", "change-brief.md") }}` and treat all settled points there as the source of truth. Ask follow-up questions only when information required to create coherent OpenSpec artifacts is still missing.
4. Before creating OpenSpec artifacts, use `Change Key` as the default OpenSpec change name. If the clarified scope shows that a better kebab-case name is needed, you may adjust it while keeping the same intent, but explicitly state the final chosen change name.
5. Once the change name is decided, create and switch to the corresponding feature branch. Prefer branching from `origin/main` when it is available; otherwise branch from the current local branch or `HEAD`. Use `feature/<change-name>` as the base form, and reuse an existing matching branch when appropriate.
6. Create the OpenSpec change under `openspec/changes/<change-name>/` inside `openspec_root` from `<tooling-context>`. If the OpenSpec CLI is available, prefer `openspec new change "<change-name>"`, then inspect `openspec status --change "<change-name>" --json` to learn the artifact graph, `applyRequires`, and schema-specific requirements.
7. When the OpenSpec CLI can provide artifact instructions, follow `openspec instructions <artifact-id> --change "<change-name>" --json` for each ready artifact, read dependency artifacts before writing the next one, apply the returned template and rules as constraints, and continue until every implementation-required artifact is complete. If the CLI is unavailable, create the equivalent scaffold manually and ensure at minimum that `.openspec.yaml`, `proposal.md`, `design.md`, `tasks.md`, and any required delta specs under `specs/` exist and are coherent.
8. Before writing the design, apply a boundary-first architecture pass: identify the owning module for each behavior change, prefer deep modules over thin pass-through layers, capture the key interface and data-flow decisions, record tradeoffs and risks explicitly, and define the validation strategy that downstream implementation should follow.
9. Write the generated OpenSpec artifacts in the language recorded in `change-brief.md` under `Requested Language`, or in `{{ setting("language", "name") }}` if no explicit request was recorded.
10. Even if frontend automation cannot be made sufficient, do not define a manual verification guide, checklist, or report as an OpenSpec deliverable or task. If needed, record only the coverage gap or verification scope there; the final manual verification document for this operation is created only in the later `manual-verification` step.
11. Do not pin references or creation targets to the current workspace. Treat `openspec_root` from `<tooling-context>` as the project root and use the `openspec/` directory under it. Use `active_projects` from `<tooling-context>` as the configured target project set when the work spans multiple projects.
12. After artifact creation, report the final `change_name` and `change_path` in `spec-plan.md`. Only if User explicitly asked to review or approve the artifacts before implementation, ask for confirmation and wait. Otherwise, proceed without waiting.
