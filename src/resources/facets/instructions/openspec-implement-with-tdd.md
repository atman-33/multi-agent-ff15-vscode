# Implement With TDD — Instructions

1. Read `{{ output("spec-planning", "latest", "spec-plan.md") }}` and use the frontmatter `change_name` as the OpenSpec change name. If `change_path` is present, use it first; otherwise use `openspec/changes/<change_name>/` under `openspec_root` from `<tooling-context>`. If `change_name` or `change_path` cannot be resolved, report the step as blocked.
2. Announce which change you are using, then inspect its current status. If the OpenSpec CLI is available, prefer `openspec status --change "<change-name>" --json` and `openspec instructions apply --change "<change-name>" --json` so you can read the schema, progress, tasks, and every concrete context file path before coding.
3. Read every context artifact that defines the change. At minimum, use the available proposal, specs, design, and tasks files. If the CLI provides `contextFiles`, treat that list as authoritative rather than assuming fixed file names.
4. Before editing code, confirm that the public interface changes and the next behavior to prove are clear. If an essential interface or behavior remains ambiguous, pause and ask for clarification instead of guessing.
5. Implement in thin vertical slices, not horizontal batches. Start from one observable behavior that can be verified through a public interface, add or update one focused failing test or equivalent executable check, make the minimum code change to pass it, then move to the next slice.
6. Follow red-green-refactor strictly. Never refactor while RED. Keep tests behavior-oriented, prefer end-to-end or boundary-facing checks over implementation-coupled tests, and avoid speculative code for future tasks.
7. After each completed slice, update the corresponding task entry immediately. If the tasks artifact uses checkboxes, change `- [ ]` to `- [x]` as soon as the slice is actually complete. If part of the change cannot be automated, record the exact coverage gap and keep the rest of the slice test-backed.
8. Keep going until the implementation-required tasks are done or you hit a blocker. If the work reveals a design problem or artifact inconsistency, stop, explain the issue concretely, and recommend whether implementation or the change artifacts should be updated next.
